      Redux middleware 详解

      kuailoy 2020-01-30 react redux javascript

      # Middleware 的由来

      Redux 本身是一个同步的单向数据流，比如下图： 如果想在 dispatch 到达 reducer 之前，进行一些操作，比如打印每一个 action 信息来调试，或者希望 dispatch 或 reducer 拥有异步请求的功能，就得去修改 dispatch 或 reducer 的实现。

      面对多样的业务场景，单纯地修改 dispatch 或 reducer 代码不具有普适性，我们需要一个可以组合的、自由插拔的插件机制。

      Redux 借鉴了 Express 和 Koa 的 middleware 思想，提供的是位于 action 被发起之后，到达 reducer 之前的扩展点，实际上 middleware 是为了增强 dispatch 而出现的。

      # middleware 机制及 redux 实现

      下面通过对源码分析，来看一下 middleware 是怎么写的，以及 Redux 是如何让 middleware 串联起来的。

      # applyMiddleware 方法

      Redux 提供了 applyMiddleware 方法来加载 middleware，我们先来看看 applyMiddleware 是如何使用的： 调用 applyMiddleware 创建一个 store

      const store = applyMiddleware(mid1, mid2, mid3, ...)(createStore)(reducer, initialState);

      这里用到了函数式编程中的柯里化(currying)，进行层层调用

      再来看一下 Redux 中 applyMiddleware 如何实现，源码如下， 把调用时传的参数对应起来，就比较清晰了，我在源码中加了一些注释

      import compose from './compose'; // compose 方法，后面会详解

export default function applyMiddleware(...middlewares) {
  // 第一层调用，middlewares对应第一个括号中的[mid1, mid2, mid3, ...]
  return (next) => (reducer, initialState) => {
    // 第二层调用：next对应createStore方法
    // 第三层调用：接收到reducer，intialState
    let store = next(reducer, initialState); // 通过createStore创建store
    let dispatch = store.dispatch; // 拿到dispatch方法
    // 11 - 17行 后面注解
    let chain = [];
    const middlewareAPI = {
      getState: store.getState,
      dispatch: (action) => dispatch(action),
    };
    chain = middlewares.map((middleware) => middleware(middlewareAPI));
    dispatch = compose(...chain)(store.dispatch);
    // 更新dispatch方法，返回store
    return {
      ...store,
      dispatch,
    };
  };
}

      再来看一个 logger middleware 的实现

      export default (store) => (next) => (action) => {
  console.log('dispatch:', action);
  next(action);
  console.log('finish:', action);
};

      对照 applyMiddleware 的代码

      let chain = [];
// 将dispatch和getState方法保存到middlewareAPI变量
const middlewareAPI = {
  getState: store.getState,
  dispatch: (action) => dispatch(action), // 这个dispatch执行时，已经是最新的dispatch
};
// 下面这行代码中，首先让每个middleware带着middlewareAPI这个参数分别执行一遍
// 这时候我们得到chain数组[f1, f2, f3, ...],里面保存的是middleware第二层函数
chain = middlewares.map((middleware) => middleware(middlewareAPI));
dispatch = compose(...chain)(store.dispatch);

      第 5 行的 middlewareAPI 中的 dispatch 为什么要用匿名函数包裹一层呢？ 因为 applyMiddleware 执行完以后，dispatch 是改变后的 dispatch 而 middlewareAPI 是在 applyMiddleware 执行过程中分发到各个 middleware 的 为了保证 middlewareAPI 中的 dispatch 是最新的，这里必须用匿名函数包裹一层

      可以做如下的代码抽象，帮助理解

      let a = () => 1;
let b = () => 2;
// 采取包裹和不包裹两种写法
const obj = {
  a: () => a(),
  b,
};
a = () => 111;
b = () => 222;

// 调用时，a是更新后的函数，b是原函数
obj.a(); // 111
obj.b(); // 2

      然后我们来看最精华的一行代码

      dispatch = compose(...chain)(store.dispatch);

      compose 是函数式编程中的组合，他将 chain 中所有匿名函数从右到左依次执行 假设 chain 中有 3 个函数，[f1, f2, f3], 调用 compose 后等同于

      dispatch = f1(f2(f3(store.dispatch))))

      当我们调用新的 dispatch 时，每一个 middleware 就依次执行了

      # middleware 中的 next 和 dispatch

      那么在 middleware 中调用 store.dispatch，和调用 next 有什么区别呢？

      在 compose 执行之后，middleware 函数层层嵌套，next 就是内层 middleware 的返回值，最后一层 middleware 的 next，就是原生的 store.dispatch

      而在 middleware 函数中调用的 store 其实是是传入的 middlewareAPI，上面解释过，store.dispatch 由于匿名函数的包裹，执行时是最新的 dispatch

      所以在 middleware 中调用 store.dispatch，作用和在外面调用是一样的，将重新进入第一个 middleware 开始执行 ，而执行 next 将会进入到下一层的 middleware， 如下图：

      正常情况下，分发一个 action 时，如左图，middleware 通过 next(action)一层层处理和传递 action 到原生的 dispatch 如果在 middleware 中调用 store.dispatch 则会发生右图的情况，重新再来一遍，如果一直简单粗暴的调用 store.dispatch(action)，就形成无限循环了

      为什么要传递 store 给 middleware 呢？关于 store.dispatch 的一个应用，可以参考 Redux Thunk Middleware

      const thunk = (store) => (next) => (action) => (typeof action === 'function' ? action(store.dispatch, store.getState) : next(action));

      Redux Thunk 会判断 action 是否是函数，如果是，则执行 action，否则继续传递 action 到下一个 middleware

      一般在异步请求的场景中，我们并不希望 action 到达最终的原生 dispatch，比如上面的 thunk 这时候，传递 store.dispatch 到 action 函数中，后续的 dispatch，都可以从头开始走一遍 middleware 的流程。

      # Compose 方法实现

      Redux 中 compose 方法，巧妙的运用 reduce 将函数组合起来，其实现如下

      function compose(...funcs) {
  return funcs.reduce((a, b) => (...args) => a(b(...args)));
}

      如果有疑问的话可以用带入参数理解一下，假设 funcs = [f1, f2, f3] 执行 reduce

      第一次: a = f1 b = f2 返回值 (...args) => f1(f2(...args))

      第二次: a = (...args) => f1(f2(...args) b = f3 返回值 (...args) => f1(f2(f3(...args))) 注意第二次执行时，f3 作为 args，传入 a 函数后，又返回了一个新函数，最终 args 为新函数的 args，在 applyMiddleware 中，args 就是调用时传入的 dispatch 方法

      # 总结

      本文简单介绍了 Redux Middleware 的原理，并分析了其代码实现，大部分内容参考了《深入 React 技术栈》第 5.2 章，并结合了自己的理解，有兴趣的同学可以买到这本书学习一下。
