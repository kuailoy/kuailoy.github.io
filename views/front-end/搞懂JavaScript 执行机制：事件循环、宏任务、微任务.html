        搞懂JavaScript 执行机制：事件循环、宏任务、微任务

        kuailoy 2020-07-30 javascript

        先来看一张图，下图展示了 JavaScript 浏览器环境的简单模型（Node 环境暂时不做讨论）

        如上图所示，浏览器环境主要包括这几个部分：

        • JS 引擎（JS Engine ）
        • WEB API
        • 事件循环 (Event Loop)
        • 任务队列(Callback Queue)

        这几个玩意儿都是干嘛的呢，它们之间又有啥关系？我们可以结合这张图来一一讨论

        # 一、JavaScript 引擎（JS Engine）

        首先，要区分 **JS 引擎（JS Engine）**和 JS 运行时环境（JS Runtime Environment） 这两个概念，它们实际上是两个维度的东西

        1. JS 运行时环境是 JS 代码运行时的宿主环境，包括了上图的框框里所有的内容，通俗点说，就是 JS 代码在哪执行，是浏览器、还是 Node
        2. 运行时环境会提供一个JS 引擎，我们的 JavaScript 代码就是由这个引擎来执行和编译的，例如 Chrome 的 V8，Mozilla 的 Spidermonkey, Safari 的 nitro 等（IE ：？），它们都是按照 ECMAScript 规范实现的，从使用 JS 语言的层面上来说，它们是没有区别的（当然，实现上肯定区别）
        3. JS 引擎只是运行时环境中的一个部分，运行时环境还会提供一些 ECMAScript 标准之外的特性，例如WEB API等

        # 二、WEB API

        WEB API 就很好理解了，接触过前端的都知道，浏览器会提供事件捕获、定时器、Http 请求等 API，这些 API 不属于 EcmaScript 的语言标准，当然也就独立于JS 引擎之外

        # 三、任务队列（Callback Queue）

        1. 我们调用 Web API 产生的一些异步任务，它们的 CallBack 会被丢进任务队列里排队等着，所以叫 Callback Queue
        2. 任务队列属于队列（Queue Data Structure）这种数据结构，遵循先进先出的原则（FIFO - First in first out）
        3. 任务队列里的 Callback 什么时候执行呢，首先是异步的结果返回了，这个时间点不能确定，其次是主线程里的代码执行完了之后，会从队列里取出第一个 Callback 执行

        P.S. 队列就跟你排队买东西一样嘛，先来的先买，后来的后买，按顺序来不能乱了

        # 四、事件循环（Event Loop）

        1. 首先要明确的是，JavaScript 是单线程的，也即只有一个主线程，事件循环是存在于主线程之外的，是宿主环境的一种机制，是更为底层的东西
        2. 主线程可以简单的理解为上图中JS 引擎中的**Call Stack，**JS 代码在主线程中同步地执行，遇到异步的任务，不等待异步结果返回，而是把它的 Callback 放入任务队列里，继续执行后面的同步代码（这里的异步任务是宏任务，通常是浏览器 API 产生的，后面会详细说明）
        3. 事件循环像是一个监工的，一直在后面运行着，在主线程的同步代码执行完后，事件循环就会去检查任务队列，然后将队列中的第一个任务取出放入主线程中继续执行，直到所有任务执行完，它就继续等待新的任务

        # 4.1 JavaScript 中的同步与异步，阻塞与非阻塞

        结合事件循环，我是这么理解这几个概念的：

        同步，是指主线程按顺序执行代码，一段代码执行完成并返回结果后，再执行下一句 如果一段代码的执行需要非常长的时间，比如一个很大的 for 循环，主线程也要一直干等着它执行完，才会执行下一句，这就造成了阻塞

        异步，是主线程执行到异步任务时，不再等待它完成，而是将其暂时挂起（pending），接着执行后面的代码，这个过程中主线程一直没闲着，即非阻塞

        它们之间的关系，可以简单理解为，同步可能造成阻塞，而异步的作用是非阻塞

        # 4.2 宏任务(macrotask)与微任务(microtask)

        在 JavaScript 引入了 Promise 等概念之后，异步代码又可以区分为宏任务和微任务，同样是异步，它们之间有什么区别呢？

        下列操作会在任务队列中，分别添加宏任务和微任务

        # 产生宏任务的操作

        • setTimeout
        • setInterval
        • I/O
        • requestAnimationFrame
        • setImmediate
        • UI rendering

        # 产生微任务的操作

        • process.nextTick(Node)
        • Promise
        • Object.observe
        • MutationObserver

        假设当前主线程的执行栈中，正在执行一个任务（宏任务），当遇到 Promise 等产生微任务的操作，就会在当前宏任务中添加一个微任务，并放入执行栈的微任务队列中，在执行完同步代码后，优先执行当前宏任务的微任务

        一个宏任务可以包含多个微任务，当前宏任务中的所有微任务都执行完后，才会执行下一个宏任务

        需要说明的是，宏任务队列和微任务队列是两个不同的队列，宏任务队列在运行时环境中，就是上面说的任务队列（Callback Queue），而微任务队列则存在于当前的执行栈（Call Stack）中，从属于当前的宏任务

        下图描述了包含多个宏任务和微任务的事件循环的过程 来看一个简单的例子：

        setTimeout(function() {
  console.log(1);
});

new Promise(function(resolve, reject) {
  resolve(3);
  console.log(2);
}).then(function(val) {
  console.log(val);
});

        要点： setTimeout  产生一个宏任务， Promise  产生一个微任务 执行顺序：同步代码 -> Promise 回调 -> setTimeout 回调

        因此执行结果为：2 3 1

        包含 setTimeout和 Promise，网上有很多复杂的例子，用同样的方法分析即可
