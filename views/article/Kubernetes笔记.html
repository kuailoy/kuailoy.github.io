    Kubernetes笔记

    kuailoy 6/4/2020 docker k8s kubernetes

    # Kubernetes介绍

    • 产生 - 从单体应用到微服务 - 组件快速迭代 - 组件的部署
    • 需求 - 自动化调度、配置、监管和故障处理
    • 作用 - kubernetes为每个组件选择合适的服务器，并保证组件通信
    • 现状 - 越来越多的大公司把kubernetes作为运行应用的最佳平台，标准化了云端部署和内部部署的应用交付方式

    # kubernetes系统的需求

    • 从单体应用到微服务：微服务的扩容、部署和环境差异
    • 为应用程序提供一致的环境：不同生产机器的环境、生产环境和开发环境
    • 持续交付：DevOps和无运维 - 让同一团队参与应用的开发、部署、运维的整个生命周期，开发者、QA和运维团队彼此之间的合作需要贯穿整个流程
    • NoOps：开发者部署应用本身，无需关心基础设施，无需与运维团队交涉，而运维团队不需要处理应用程序的独特性

    kubernetes能够实现上述内容，它通过对实际硬件做抽象，将自身暴露成一个平台，允许开发者自己部署应用程序，让系统管理员聚焦于底层基础设施

    # 核心概念

    • Pod
      • 最小部署单元
      • 一组容器的集合
      • 容器共享网络
      • 生命周期是短暂的
    • Controller
      • 确保预期的pod副本的数量
      • 无状态/有状态应用部署
      • 确保所有的node运行同一个pod
      • 一次性任务和定时任务
    • Service
      • 定义一组pod的访问规则
    • 存储
    • 调度器Ingress
    • 安全机制RBAC
    • 包管理工具Helm

    # k8s集群架构组件

    • Master Node（主控节点）
      • API server: 集群统一入口，通过UI或CLI与集群进行交互，以RESTful方式，交给etcd存储
      • scheduler: 节点调度，选择容器运行的node节点
      • controller-manager: 处理集群中常规后台任务，一个资源对应一个控制器
      • etcd: 存储系统，用于保存集群相关的状态和配置
    • node （工作节点）
      • docker：容器技术
      • pod: 容器调用的最小单位, 容器的集合
      • kubelet: master派到node节点的代表，管理本机容器
      • kube-proxy: 提供网络代理，负载均衡等操作
      • fluentd: 日志的采集和查询

    # pod

    • pod是一组并置的容器，代表kubernetes中的基本构建模块
    • 同一个pod中的容器运行于相同的Network命名空间、相同的loopback网络接口，可以通过localhost通信
    • 集群中的所有pod都在同一个共享网络地址空间中，可以通过ip地址相互访问
    • 容器不应该包含多个进程，pod也不应该包含多个不需要运行在同一主机上的容器

    ​

    查看已部署pod的完整YAML

    $ kubectl get po [pod名称] -o yaml

    完整的YAML信息

    • apiVersion: v1 YAML描述文件所使用的kubernetes API版本
    • kind：pod Kubernetes对象/资源类型
    • metadata pod元数据（名称、标签和注解等）
    • spec pod规格/内容（pod的容器列表、volume等）
    • status pod及其内部容器的详细状态

    # 标签 (label)

    标签是可以附加到资源的任意键值对 使用标签可以更好的组织pod和进行分类 ​

    创建pod指定标签

    ...
metadata:
  name: kubia-manual-v2
  labels:
    creation_method: manual
    env: prod
...

    修改现有的pod标签

    # 给pod添加标签
$ kubectl label po kubia-manual creation_method=manual

# 修改现有标签，需要使用--overwrite选项
$ kubectl label po kubia-manual-v2 env=debug --overwrite

    通过标签选择器列出pod

    # 列出creation_method=manual的pod
$ kubectl get po -l creation_method=manual

# 列出包含env标签的所有pod，无论其值如何
$ kubectl get po -l env

# 列出没有env标签的pod
$ kubectl get po -l '!env'

# 其他(验证失败)
# creation_method!=manual
# env in (prod, devel)
# env notin (prod, devel)

    # 通过标签选择器将pod调度到特定节点

    给YAML文件增加nodeSelctor属性，如下

    spec:
  nodeSelector:
    gpu: "true"

    这样，创建pod时，调度器将只在包含标签gpu=true的节点中选择 ​

    使用标签分类工作节点

    # 给节点添加gpu=true标签
$ kubectl label node [nodeName] gpu=true

# 列出只包含标签gpu=true的节点
$ kubectl get nodes -l gpu=true

# 列出所有节点, 附加gpu标签值列
$ kubectl get nodes -L gpu

    将pod调度到特定节点

    $ kubectl create -f kubia-gpu.yaml

    # 注解 (annotations)

    pod和其他对象还可以包含注解，注解也是键值对 注解不能像标签一样用于对对象进行分组

    # 查看对象的注解

    $ kubectl get po [podName] -o yaml
# 或
$ kubectl describe [podName]

    添加和修改注解

    $ kubectl annotate pod [podName] mycompany.com/someannotation="foo bar"

    # 命名空间

    命名空间简单的为对象名称提供了一个作用域，不同的命名空间可以包含同名的资源

    # 列出集群中所有命名空间
$ kubectl get ns

# 查看指定命名空间的pod
$ kubectl get po --namespace kube-system
#	使用-n代替--namespace
$ kubectl get po -n kube-system

    # 创建命名空间

    从YAML文件创建命名空间： 创建一个custom-namespace.yaml文件，如下 

    $ kubectl create -f custom-namespace.yaml

    使用kubectl create namespace命令创建

    $ kubectl create namespace custom-namespace

    # 管理其他命名空间中的对象

    在其他命名空间中创建资源

    $ kubectl create -f kubia-manual.yaml -n custom-namespace

    当前上下文和当前上下文的命名空间可以通过kubectl config 命令进行更改 可以设置下列别名：

    alias kcd='kubectl config set-context $(kubectl config current-context) --namespace'default

    然后，可以使用kcd some-namespace在命名空间之间进行切换 ​

    命名空间是否隔离取决于kubernetes使用的网络解决方案 ​

    # 按名称删除pod
$ kubectl delete po kubia-gpu

# 删除多个pod
$ kubectl delete po pod1 pod2

# 使用标签选择器删除pod

    # 存活探针（liveness probe）

    Kubernetes可以通过存活探针检查容器是否还在运行，可以为pod中的每个容器单独指定存活探针，如果探测失败，kubernetes将定期执行探针并重新启动容器 ​

    Kubernetes有以下三种探测容器的机制

    • HTTP GET 探针对容器的IP地址（你指定的端口和路径）执行HTTP GET请求
    • TCP套接字探针尝试与容器建立TCP链接，成功建立则探测成功
    • Exec探针在容器内执行任意命令，并检查命令的退出状态码是0则探测成功

    # 创建基于HTTP的存活探针

    YAML相关配置

    获取崩溃容器的日志

    # 当前容器的日志
$ kubectl logs mypod

# 前一个容器的日志
$ kubectl logs mypod --previous

    通过查看kubectl describe可以了解容器重启原因 ​

    # 存活探针的附加配置

    存活探针可以增加附加配置，例如要设置初始延迟

    livenessProbe:
  httpGet:
    path: /
    port: 8080
  initialDelaySeconds: 15

    如果不设置初始延迟，通常会导致探测失败，因为程序还没准备好 ​

    # 创建有效的存活探针

    • 给探针配置特定的URL路径，并让应用从内部对重要组件执行状态检查
    • 保持探针轻量，不应消耗太多资源
    • 无须在探针中实现重试循环，kubernetes会尝试若干次
    • 存活探针失败时由节点的kubelet执行重启操作，如果节点本身崩溃，将无法执行任何操作

    ​

    # ReplicationController

    节点故障时，ReplicationController管理的pod会被重新创建 ReplicationController的工作是确保pod数量始终与其标签选择器相匹配 ReplicationController的三部分

    • label selector (标签选择器)，用于确定ReplicationController作用域中有哪些pod
    • replica count (副本个数)，指定应运行的pod数量
    • pod template (pod模板)， 用于创建新的pod副本

    其中，只有更改副本个数，会影响现有的pod 更改标签选择器和pod模板，会使现有pod脱离ReplicationController的范围 ​

    使用RepicationController的好处

    • 确保一个pod持续运行，方法是在现有pod丢失时启动一个新的pod
    • 集群节点发生故障时，会给故障节点上运行的所有pod创建替代副本
    • 能够轻松实现pod的水平伸缩

    # 创建一个ReplicationController

    定义ReplicationContrller时不要指定pod选择器，让Kubernetes从pod模板中提取它，这样YAML更简短 ​

    使用kubectl create命令

    $ kubectl create -f kubai-rc.yaml

    之后可以查看pod，来看ReplicationController是否工作
